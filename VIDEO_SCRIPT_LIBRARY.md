## VIDEO 2: 2-Minute Mental Health Screening Explainer

**Title:** "How Depression Screening Works in HDIM (2 Minutes)"  
**Format:** Animated walkthrough + on-screen graphics + patient/staff testimonials  
**Target:** Healthcare professionals wanting to understand the workflow  
**Goal:** Build confidence in the solution, address "how does this actually work?" questions

---

### Script

**[0:00-0:15] OPENING**

[Narrator voice, warm and professional]

"Depression screening in primary care sounds complicated.

But it doesn't have to be.

Let's walk through exactly how it works, step by step."

**[Show HDIM patient check-in interface on screen]**

[Animated graphics showing patient arriving at clinic]

The adaptive PHQ-9 asks only the questions that matter—not every patient answers the same questions.

It takes 90 seconds."

**[0:35-0:55] STEP 2: AUTOMATIC SCORING**

[Animation showing data flowing from tablet to system]

"Behind the scenes, HDIM analyzes the responses in real time.

It calculates a depression risk score.

More importantly, it checks the patient's history, their current diagnoses, their quality gaps.

In this case, the patient screens positive for depression—their first screen ever.

This is exactly the gap we're trying to close."

**[Show score: 12 points - Moderate depression]**

**[0:55-1:25] STEP 3: INTELLIGENT ROUTING**

[Animation showing branching pathways]

"Now here's where automation starts to matter.

Instead of a care coordinator having to manually review the result and decide what to do, HDIM does this automatically.

Severity scores determine routing:

- Low depression risk? No alert. Normal workflow.
- Moderate depression risk? Task created for care coordinator.

All in the next 30 seconds.

The patient hasn't left the waiting room yet, and the entire clinical team has been notified."

**[1:25-1:45] STEP 4: DOCUMENTATION & GAPS**

[Animation showing EHR update in real time]

"HDIM automatically documents the screening in the patient's EHR as a completed HEDIS element.

It also creates a care gap: 'Depression screening positive—no documented treatment plan.'

Your quality team now sees this patient counted in real time in their quality dashboard.

She's no longer a hidden depression case. She's a documented gap that can be closed."

**[Show quality dashboard updating in real time]**

**[1:45-2:00] CLOSING**

[Narrator, confident]

"From check-in to documentation to gap creation: 3 minutes.

No manual forms. No delayed data entry. No lost results buried in an inbox.

That's the power of automation in depression screening.

Ready to see this in your environment?

Let's talk."

- Fades up during CTA

- Text on screen: "Schedule a Demo" + [URL]

## VIDEO 4: 3-Minute FHIR-Native Architecture Overview

**Title:** "Why FHIR-Native Architecture Matters (3 Minutes)"  
**Format:** Technical explainer + architecture diagrams + comparison  
**Target:** IT leaders, CIOs, healthcare IT architects  
**Goal:** Establish technical credibility, show advantage of native architecture

---

### Script

**[0:00-0:20] OPENING**

[Technical but approachable tone]

"Let's talk about something that IT leaders ask us about constantly:

'Why does HDIM matter more if it's FHIR-native?'

Great question. The answer has to do with interoperability, implementation speed, and long-term flexibility.

Stick with me. I'll make this make sense."

**[Show FHIR logo on screen]**

**[0:20-0:50] WHAT IS FHIR?**

[Animated explainer]

"FHIR stands for Fast Healthcare Interoperability Resources.

It's an HL7 standard designed to make healthcare data portable and interoperable.

Instead of creating custom interfaces between each system, FHIR creates a standard format that every system can understand.

Think of it like a universal language for healthcare data.

Without FHIR, every system has its own language. Integrating them requires custom translation layers.

With FHIR, they all speak the same language."

**[Show animated graphic: Without FHIR (many arrows, complex), With FHIR (clean interfaces, simple)]**

**[0:50-1:30] FHIR-NATIVE VS. RETROFITTED**

[Side-by-side architecture diagrams]

"Here's the key distinction:

**FHIR-Retrofitted Architecture:**

Traditional healthcare tools were built before FHIR existed. They use HL7 v2, proprietary databases, old integration methods.

When FHIR came out, they added a FHIR layer on top of their existing system.

Result: You're translating between old architecture (internal) and new standard (external).

Implementation: 18-24 months
Cost: $200-500K in consulting
Complexity: High (custom interfaces, ongoing maintenance)

**FHIR-Native Architecture:**

Some tools, like HDIM, were built from the ground up on FHIR.

Your data is stored natively in FHIR format.

Your APIs are FHIR-compliant.

Your integrations are FHIR-compliant.

Result: No translation layers. Direct data flow.

Implementation: 8-12 weeks
Cost: Minimal (standard APIs, plug-and-play)
Complexity: Low (standard interfaces, easy to maintain)"

**[Show architectural diagrams, highlighting data flow differences]**

**[1:30-2:00] REAL-WORLD EXAMPLE: EPIC INTEGRATION**

[Animated workflow]

"Let's use Epic as an example.

With a FHIR-retrofitted system:
1. Your Epic data leaves your environment via custom HL7 v2 interface
2. It's translated to the vendor's proprietary format
3. The vendor translates it to FHIR for external consumption
4. You integrate with Epic via FHIR API
5. Epic sends FHIR data back to the vendor
6. The vendor translates it back to proprietary format
7. The vendor translates it back to HL7 v2
8. It arrives in Epic

Seven steps. Multiple translation layers. Each is a failure point.

With HDIM (FHIR-native):
1. Your Epic data reaches HDIM via FHIR API
2. HDIM processes it natively (no translation)
3. HDIM sends results back to Epic via FHIR API
4. Done

Three steps. No translation. No failure points."

**[Show step-by-step process animation for each approach]**

**[2:00-2:40] THE BUSINESS IMPACT**

[Financial and operational graphics]

"Here's what this means in practice:

**Speed:**
- FHIR-retrofitted: 18-24 months to production
- FHIR-native: 8-12 weeks to production
- Advantage: FHIR-native is 4X faster

**Cost:**
- FHIR-retrofitted: $200-500K in consulting + your internal IT time
- FHIR-native: Minimal consulting (it's plug-and-play) + your internal IT time (reduced by 70%)
- Advantage: FHIR-native costs 60-70% less

**Maintenance:**
- FHIR-retrofitted: Ongoing custom interface maintenance
- FHIR-native: Standard FHIR maintenance (updates handled by vendor)
- Advantage: FHIR-native requires 80% less maintenance effort

**Flexibility:**
- FHIR-retrofitted: Adding new systems requires custom interfaces
- FHIR-native: Adding new systems plugs into existing FHIR infrastructure
- Advantage: FHIR-native scales without multiplying complexity

The compounding effect: FHIR-native is not just faster to implement, it's easier to maintain and cheaper to extend."

**[Show comparison table on screen: Speed, Cost, Maintenance, Flexibility]**

**[2:40-3:00] CLOSING**

[Direct and confident]

"If you're evaluating healthcare software, FHIR-native architecture should be on your checklist.

It's the difference between a 12-week implementation and a 24-month implementation.

It's the difference between standard APIs and custom interfaces.

It's the difference between future-ready and legacy.

Want to understand HDIM's FHIR architecture in more detail?

Let's do a technical deep-dive with your IT team."

**[End with HDIM logo + CTA]**

## PRODUCTION SPECIFICATIONS

### All Videos

**Format:** 16:9 widescreen (1920x1080 or higher)  
**Codec:** H.264 MP4 (YouTube, LinkedIn, web)  
**Frame Rate:** 30fps (standard for B2B content)  
**Color Grade:** Professional color correction, consistent across all videos  
**Audio Levels:** Normalized to -14dB (YouTube recommended)  
**Captions:** Burned-in captions (white text, drop shadow) + SRT file for accessibility

### Text Overlay Standards

- Font: Helvetica Neue or similar (professional, readable)
- Size: Large enough to read on mobile (18pt minimum)
- Color: White with drop shadow (readable on any background)
- Duration: 3-5 seconds per text block
- Animation: Fade in/fade out (no jarring cuts)

### Thumbnail Design

- HDIM logo in corner (brand consistency)
- Large text with main value prop (e.g., "Depression Screening in 90 Days")
- Color contrast (stand out in YouTube feed)
- No faces in thumbnails (B2B best practice)

### Distribution Specifications

**YouTube:**
- Title: [Video Title] | HDIM (exact match script title)
- Description: 300-400 words including key points, links, CTA
- Tags: 10-15 tags (depression screening, HDIM, risk stratification, healthcare, etc.)
- Thumbnail: Custom thumbnail (follow design specs above)
- Playlist: Add to "Mental Health & Quality Measurement" playlist
- Link: Prominently link to webinar registration or discovery call scheduling

**LinkedIn:**
- Post text: 2-3 sentences + 1 CTA
- Video player: Upload native video (higher engagement than YouTube links)
- Hashtags: #HealthcareIT #QualityMeasurement #MentalHealth #FHIR
- CTA button: "Learn More" or "Watch Webinar"

**Website/Landing Pages:**
- Host videos on Vimeo (faster loading than YouTube)
- Embed with autoplay (muted) on landing pages
- Embed with CTA overlay: "Register for Webinar" after 30 seconds
