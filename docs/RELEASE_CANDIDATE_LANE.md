# Release Candidate Lane

This repo uses a **release-candidate (RC) lane** so we can fully validate locally, capture media, and only then rely on GitHub Actions for final processing.

## Branching Model

- `master`
  - Integration branch.
  - Must remain releasable (or close to it).
- `release/<version>`
  - Release candidate branch.
  - Naming: `release/vX.Y.Z-rcN` (example: `release/v2.7.1-rc1`).
  - Only accepts changes that are explicitly within the RC scope.
- `hotfix/<version>`
  - Optional: urgent fixes after a release is tagged.

## Version Scope: What “Defined” Means

For every RC version `vX.Y.Z-rcN`, we must have:

1. A scope file:
   - `docs/releases/<version>/SCOPE_<version>.md`
2. Release docs generated from templates:
   - `docs/releases/<version>/RELEASE_NOTES_<version>.md`
   - `docs/releases/<version>/UPGRADE_GUIDE_<version>.md`
   - `docs/releases/<version>/VERSION_MATRIX_<version>.md`
   - `docs/releases/<version>/KNOWN_ISSUES_<version>.md`
   - `docs/releases/<version>/PRODUCTION_DEPLOYMENT_CHECKLIST_<version>.md`
3. Validation output:
   - `docs/releases/<version>/validation/*` (generated by scripts under `scripts/release-validation/`)
4. Media plan + artifacts:
   - `docs/releases/<version>/media/*` (screenshots + videos)

## Creating an RC

1. Cut an RC branch from `master`:
```bash
git switch master
git pull --rebase
git switch -c release/vX.Y.Z-rc1
```

2. Generate release docs (creates `docs/releases/<version>/...`):
```bash
./scripts/release-validation/generate-release-docs.sh vX.Y.Z-rc1
```

3. Define scope:
   - Edit `docs/releases/vX.Y.Z-rc1/SCOPE_vX.Y.Z-rc1.md` and list:
     - included PRs/commits
     - excluded items (explicitly)
     - acceptance criteria

## RC Stages (What Each Version Is Allowed To Contain)

This keeps releases predictable and prevents “RC drift”.

- `vX.Y.Z-rc1`
  - Primary goal: make the demo + release workflow deterministic.
  - Allowed: demo seeding/validation fixes, local CI improvements, scope docs, MCP/tooling stabilization, must-fix bug fixes.
  - Not allowed: new product surface area unless explicitly required for the demo readiness gate.
- `vX.Y.Z-rc2` (and higher)
  - Primary goal: polish and fix bugs found during recording/validation.
  - Allowed: targeted bug fixes, performance tuning, copy/UI polish for recorded flows, release docs updates.
  - Not allowed: architecture rewrites, broad dependency upgrades, new features.
- `vX.Y.Z` (tag)
  - Only after: `./scripts/ci/local-ci.sh pr` passes, demo readiness gate passes, and media is captured.

## Scope Rules (to avoid RC drift)

- Allowed:
  - bug fixes required for demo/readiness gates
  - release documentation fixes
  - deterministic demo seeding and validation fixes
  - performance fixes needed for demos (clear measurement required)
- Not allowed:
  - new features without explicit scope update + re-validation
  - refactors without clear user-impact and rollback plan
  - dependency upgrades unless required for stability/security

## Promotion Criteria (RC -> Release)

All must be true:

- Local PR gate passes:
  - `./scripts/ci/local-ci.sh pr`
- Demo readiness gate passes (see `docs/DEMO_READINESS_GATE.md`).
- Media captured for differentiators (see `docs/media/CAPTURE_CHECKLIST.md`).
- Release validation reports exist under:
  - `docs/releases/<version>/validation/`
- Tagging checklist completed:
  - `./scripts/release-validation/validate-git-status.sh <version>`

When ready to cut the release tag:
```bash
git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z
```
