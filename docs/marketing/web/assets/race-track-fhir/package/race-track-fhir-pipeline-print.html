One canonical patient journey shown at three detail levels for non-technical, mixed, and technical audiences.

Built from real-world HIE operating patterns (Healthix, HealthInfoNet) and modern identity strategy (Verato).

View 1: Non-Technical

Customer Data Sources

Customer FHIR server and CDR provide the starting point.

Standards-Based Processing

Events move in real time through standards-based workflows.

Care Decision Checkpoint

Clinical context is applied before action recommendation.

Action Delivered

Standardized actions return to care teams and systems.

Presenter line: Patients follow one continuous workflow. Data moves with them end-to-end, so onboarding does not require custom interfaces.

View 2: Mixed

Customer Domain

Customer FHIR Server and Customer CDR/EHR.

Interoperability Boundary

FHIR R4 normalization from customer data into an internal FHIR longitudinal model.

Pipeline Core

Kafka topics: hdim.patient.events.v1, hdim.care_gap.events.v1, hdim.quality.events.v1, hdim.actions.events.v1.

Services + Cache

Patient/Care Gap/Quality Measure/Event services with Redis patient snapshot, rule context, and idempotency keys.

Presenter line: Customer FHIR/CDR data crosses a standards boundary into internal FHIR, is processed through Kafka + services, then returned as FHIR-native actions.

View 3: Technical

Producers + Canonicalization

Customer FHIR and CDR feeds create canonical events with tenant context controls.

Kafka Topology

Versioned topics and consumer groups with partition strategy, hdim.patient.events.retry.v1, and hdim.patient.events.dlq.v1.

Redis Roles

Patient Snapshot Cache, Rule Context Cache, Idempotency Keys.

Delivery + Observability

FHIR-native write-back plus consumer lag, distributed tracing, and PHI-safe logging boundaries.

Presenter line: This is deterministic producer-consumer processing with explicit cache roles, idempotent action publishing, retry/DLQ controls, and tenant-safe observability.
