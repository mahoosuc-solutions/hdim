## Overview

Prove every MCP Edge tool works against the real HDIM gateway with clinically valid synthetic patient data. Current tests use demo mode fixtures — this design adds full round-trip integration tests that exercise tool handlers → HTTP clients → gateway → FHIR/CQL/quality/care-gap services.

Synthetic patients are generated by Synthea (MITRE, peer-reviewed) with hand-crafted overlays for care gaps, quality measures, and CQL targets. The phenotype data doubles as external-facing clinical validation material.

## Architecture Decision

**Approach A: Sidecar-Internal Live Tests.**

Each sidecar gets a `__tests__/live/` directory. Tests create the Express app via `createApp()` with `HDIM_DEMO_MODE=false`, then use supertest to call tools through MCP protocol. Tool handlers make real HTTP calls to the running gateway.

### Why This Approach

- Same supertest pattern developers already know
- Tests validate the full stack: MCP protocol → router → tool handler → HTTP client → gateway
- Each sidecar owns its own live tests
- No new packages or test runners to maintain
- Graceful skip when gateway is unavailable (local dev without stack)

## Synthetic Patient Phenotypes

### Generation Pipeline

Synthea (MITRE) generates base FHIR R4 Bundles. Hand-crafted overlays add platform-specific test targets.

```
test-data/synthetic-patients/
├── synthea/                    # Synthea config per phenotype
├── bundles/                    # Generated FHIR R4 Bundles (committed)
├── overlays/                   # Care gaps, CQL targets, measure data
├── manifest.json               # Phenotype → bundle → overlay → expected outcomes
├── load-test-patients.js       # Reads bundles + overlays → POSTs to gateway
└── generate.sh                 # Wraps Synthea CLI (bundles committed, CI doesn't need Synthea)
```

### Overlay Structure

Each overlay adds platform-specific data on top of Synthea's clinically valid base:

- **Care gaps:** Open/closed/overdue gaps linked to specific quality measures
- **CQL targets:** Known expected outcomes for `cql_evaluate` (compliant/non-compliant)
- **Quality measure periods:** Measurement year boundaries for `measure_evaluate`
- **CDS hook triggers:** Conditions that should fire patient-view CDS cards

## Test Data Discovery

Tests don't hardcode UUIDs. After seeding, a setup phase discovers resource IDs:

1. `demo_seed` via platform sidecar → populates gateway
2. `load-test-patients.js` → loads Synthea bundles + overlays into FHIR server
3. `fhir_search(Patient)` → discovers patient IDs per phenotype
4. `fhir_search(Observation, { patient })` → discovers observation IDs
5. Same for Condition, Encounter, MedicationRequest, Immunization
6. Results written to `/tmp/live-test-context.json`
7. All subsequent tests read from shared context

```js
module.exports = {
  LIVE_TEST_DEFAULTS: {
    tenantId: 'demo',
    gatewayUrl: process.env.HDIM_BASE_URL || 'http://localhost:18080',
  },

  async isGatewayReachable(url) { /* fetch /actuator/health, return boolean */ },
  async isDockerReachable() { /* exec docker info, return boolean */ },

  readTestContext() { /* reads /tmp/live-test-context.json */ },
  writeTestContext(ctx) { /* writes /tmp/live-test-context.json */ },

  skipUnlessLive(checkFn) {
    /* returns beforeAll that skips all tests if checkFn returns false */
  }
};
```

### Skip Mechanism

Each live test file starts with:

```js
const { skipUnlessLive, isGatewayReachable, LIVE_TEST_DEFAULTS } = require('hdim-mcp-edge-common/lib/live-test-helpers');

beforeAll(async () => {
  const reachable = await isGatewayReachable(LIVE_TEST_DEFAULTS.gatewayUrl);
  if (!reachable) {
    console.warn('Gateway unreachable — skipping live tests');
    // Jest conditional skip
  }
});
```

### Test File Structure

```
mcp-edge-platform/__tests__/live/
  platform-tools.live.test.js          — 8 platform tool round-trips

mcp-edge-clinical/__tests__/live/
  setup.live.test.js                   — seed + discover IDs (runs first)
  fhir-tools.live.test.js             — fhir_read, fhir_search, fhir_create
  patient-tools.live.test.js          — patient_summary, timeline, risk, list
  care-gap-tools.live.test.js         — care_gap_list, identify, stats, population, close
  quality-tools.live.test.js          — measure_evaluate, results, score, population
  cql-tools.live.test.js             — cql_evaluate, batch, result, libraries
  cds-tools.live.test.js             — cds_patient_view, health_score, pre_visit_plan

mcp-edge-devops/__tests__/live/
  devops-tools.live.test.js           — docker_status, logs, restart, etc.
```

### Naming Convention

`*.live.test.js` — Jest `testPathPattern` includes/excludes these.

### Execution Order

1. `setup.live.test.js` runs first (`--runInBand` for live tests)
2. Read-only tests run in parallel
3. Write tests (`fhir_create`, `fhir_bundle`, `care_gap_close`) run last

### Response Contract Validation

Every live test validates:

| Assertion | What It Proves |
|-----------|---------------|
| `res.status === 200` | MCP protocol success |
| `result.content[0].type === 'text'` | MCP response format |
| `JSON.parse(text).ok === true` | Gateway returned success |
| `JSON.parse(text).status === 200` | HTTP 200 from gateway |
| `JSON.parse(text).data` defined | Response has a body |
| Resource-specific shape checks | Correct FHIR resourceType, expected fields |

### Phenotype-Specific Assertions

Beyond response contracts, tests validate clinical correctness against `manifest.json` expected outcomes:

- `t2dm-managed`: `measure_evaluate` → all measures compliant, `care_gap_list` → 0 gaps
- `t2dm-unmanaged`: `care_gap_identify` → retinopathy + nephropathy gaps, `cql_evaluate(HbA1c-Control)` → non-compliant
- `chf-polypharmacy`: `patient_risk` → high risk score, medication count ≥ 8
- `preventive-gaps`: `care_gap_list` → 3+ open gaps
- `healthy-pediatric`: `patient_summary` → no active conditions, immunizations current
- `multi-chronic-elderly`: `health_score` → low score, 3+ active conditions

## DevOps Live Tests

DevOps tools use Docker socket, not the gateway. Live tests target the running MCP Edge Docker Compose stack:

- `docker_status` → sees `mcp-edge-platform`, `mcp-edge-clinical`, `mcp-edge-devops` containers
- `docker_logs` → retrieves real log output from a known container
- `service_dependencies` → returns dependency graph for MCP Edge services
- `compose_config` → reads `docker-compose.mcp-edge.yml`
- `build_status` → reports container image info
- `docker_restart` → tested last, targets only the devops container itself (safe, self-restart)

## CI Integration

### New Job in `mcp-edge-ci.yml`

```yaml
live-tests:
  name: MCP Edge Live Integration
  runs-on: ubuntu-latest
  timeout-minutes: 15
  if: github.ref == 'refs/heads/master' || startsWith(github.ref, 'refs/heads/release/')

  steps:
    - Checkout
    - Setup Node.js 20
    - Setup Java 21 (for gateway)
    - Start full stack: docker compose -f docker-compose.demo.yml up -d
    - Wait for gateway health: poll /actuator/health (max 120s)
    - Install npm deps: npm ci
    - Load synthetic patients: node test-data/synthetic-patients/load-test-patients.js
    - Run live tests: npx jest --testPathPattern=live --runInBand --no-coverage
    - Collect results + upload artifact
    - Teardown: docker compose down
```

### Environment Variables

```
HDIM_BASE_URL=http://localhost:18080
HDIM_DEMO_MODE=false
LIVE_TESTS=true
MCP_EDGE_ENFORCE_ROLE_AUTH=true
```

### Branch Gating

- **master + release/***: Live tests required to pass
- **feature branches**: Live tests skipped (no stack available)
- **workflow_dispatch**: Manual trigger for ad-hoc runs

## Testing Strategy

TDD with tests written before implementation:

1. **Shared utilities** — live-test-helpers with skip mechanism, context read/write
2. **Synthea pipeline** — generate.sh, manifest.json, verify bundles are valid FHIR
3. **Overlay loader** — load-test-patients.js, verify bundles + overlays POST successfully
4. **Platform live tests** — 8 tools against real gateway
5. **Clinical setup** — seed + discover IDs
6. **Clinical live tests** — 25 tools across 7 test files, phenotype-specific assertions
7. **DevOps live tests** — 7 tools against Docker Compose stack
8. **CI workflow** — add live-tests job, verify gating logic
9. **Clinical validation docs** — phenotype profiles, methodology, validation matrix
