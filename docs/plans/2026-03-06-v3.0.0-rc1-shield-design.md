**Inbound (C-CDA Document):**
```

  → IHE XDS.b Provide & Register [ITI-41]

  → healthix-adapter-service (C-CDA parse → FHIR Bundle)
  → Kafka: external.healthix.documents
  → cdr-processor-service (clinical data repository)
  → care-gap-service + quality-measure-service
```

**Outbound (HDIM → Hospital/Payer):**
```
care-gap-service (gap identified)
  → Kafka: external.hdim.caregaps
  → hedis-adapter-service (HEDIS/MIPS/STAR calculation)
  → hedis CQL engine (external :3333)
  → Kafka: external.hedis.calculations
  → WebSocket bridge → hedis dashboard (real-time)
  → corehive-adapter-service (AI prioritization, de-identified)
  → CoreHive AI (:3067) → ranked priorities
```

**Outbound (FHIR Bulk Export for OR-HIE):**
```
Oregon Health Authority (OHA) request
  → Kong Gateway [mTLS + API key]
  → fhir-service (:8085) [FHIR Bulk Data $export]
  → async job → NDJSON files
  → IHE XCA Cross-Community Query [ITI-38/39]
```

## 4. IHE Interface Specifications

### 4.1 XDS.b — Cross-Enterprise Document Sharing

**Purpose:** Exchange C-CDA, DICOM, and other clinical documents between HDIM, Healthix, and Oregon HIE participants.

**HDIM Roles:**
- **Document Source** — Push new documents (care gap reports, quality summaries) to the HIE registry
- **Document Consumer** — Query and retrieve documents from Healthix and other HIE participants

**Transactions:**

| ITI | Name | Direction | Protocol | Endpoint |
|-----|------|-----------|----------|----------|
| ITI-18 | Registry Stored Query | HDIM → Healthix | SOAP/REST | `/ihe/xds/query` |
| ITI-41 | Provide & Register | HDIM → Healthix | SOAP/REST | `/ihe/xds/provide` |
| ITI-42 | Register Document Set | HDIM → Registry | SOAP | `/ihe/xds/register` |
| ITI-43 | Retrieve Document Set | HDIM → Healthix | SOAP/REST | `/ihe/xds/retrieve` |

**Document Types Supported:**
- C-CDA R2.1 (Continuity of Care, Discharge Summary, Referral Note)
- FHIR R4 DocumentReference
- DICOM (via WADO-RS reference)
- Quality measure reports (QRDA Category I/III)

### 4.2 PIXv3 — Patient Identifier Cross-Referencing

**Purpose:** Resolve patient identifiers across HDIM, Healthix (Verato MPI), hospital MRNs, and Oregon HIE participant IDs.

**HDIM Role:** Consumer — query Healthix MPI for cross-references.

| ITI | Name | Direction | Protocol |
|-----|------|-----------|----------|
| ITI-44 | Patient Identity Feed | HDIM → Healthix MPI | HL7v3/SOAP |
| ITI-45 | PIX Query | HDIM → Healthix MPI | HL7v3/SOAP or FHIR $match |
| ITI-46 | PIX Update Notification | Healthix MPI → HDIM | HL7v3/SOAP |

**Identifier Domains:**
- HDIM internal patient UUID
- Hospital MRN (per facility OID)
- Oregon Medicaid ID
- SSN (hashed, never transmitted in clear)
- Healthix Verato Enterprise ID (EUID)

### 4.3 PDQv3 — Patient Demographics Query

**Purpose:** Allow OR-HIE participants to search HDIM's patient registry by demographics.

**HDIM Role:** Supplier — expose patient demographics query endpoint.

| ITI | Name | Direction | Protocol |
|-----|------|-----------|----------|
| ITI-47 | Patient Demographics Query | External → HDIM | HL7v3/SOAP or FHIR |
| ITI-21 | Patient Demographics Query (v2) | External → HDIM | HL7 v2.5 QBP^Q22 |

### 4.4 XCA — Cross-Community Access

**Purpose:** Federated document query/retrieve across Oregon HIE communities (HDIM ↔ Healthix ↔ other HIEs).

| ITI | Name | Direction | Protocol |
|-----|------|-----------|----------|
| ITI-38 | Cross Gateway Query | HDIM → remote communities | SOAP |
| ITI-39 | Cross Gateway Retrieve | HDIM → remote communities | SOAP |

### 4.5 ATNA — Audit Trail and Node Authentication

**Purpose:** HIPAA-compliant audit logging and mutual TLS authentication for all IHE transactions.

**Requirements:**
- All IHE nodes authenticated via mTLS (X.509 certificates)
- All transactions generate RFC 3881 / DICOM Supplement 95 audit events
- Audit events forwarded to centralized ATNA Audit Record Repository
- Syslog (RFC 5424) transport with TLS

## 5. FHIR High-Throughput Architecture

### 5.1 Performance Targets (Oregon HIE Scale)

| Metric | Target | Rationale |
|--------|--------|-----------|
| Sustained throughput | 500+ FHIR messages/sec | OR-HIE peak: ~300 msg/sec across 62 hospitals |
| Read latency (p99) | <200ms | Clinical workflow SLA |
| Write latency (p99) | <500ms | Document registration SLA |
| Bulk export throughput | 100K resources/min | Population health reporting |
| Concurrent connections | 1000+ WebSocket | Real-time dashboard feeds |
| Availability | 99.95% | Healthcare SLA |

### 5.2 Optimization Strategy

**Connection Pooling:**
```yaml
# Per-adapter HTTP client pool
spring:
  codec:
    max-in-memory-size: 10MB
external:
  healthix:
    connection-pool:
      max-connections: 200
      max-connections-per-route: 50
      connection-timeout-ms: 3000
      socket-timeout-ms: 10000
      keep-alive-ms: 30000
```

**Kafka Tuning for Throughput:**
```yaml
spring:
  kafka:
    producer:
      batch-size: 32768        # 32KB batches
      linger-ms: 10            # 10ms linger for batching
      buffer-memory: 67108864  # 64MB buffer
      compression-type: lz4    # Fast compression
      acks: 1                  # Leader ack only for throughput
    consumer:
      fetch-min-size: 1048576  # 1MB min fetch
      fetch-max-wait-ms: 500   # 500ms max wait
      max-poll-records: 500    # Batch consumption
```

**FHIR Response Caching (Non-PHI):**
- Cache `CapabilityStatement`, `StructureDefinition`, `ValueSet`, `CodeSystem`
- Redis TTL: 1 hour for metadata, 5 min for search results (non-PHI only)
- Cache invalidation via Kafka topic `fhir.cache.invalidation`

**Async Processing Pipeline:**
- HL7 v2.5 ADT messages: Kafka consumer → async FHIR transformation → async persistence
- C-CDA documents: Kafka consumer → async XML parse → async FHIR Bundle creation → async storage
- Care gap detection: event-driven via Kafka (no synchronous chain)

### 5.3 Backpressure & Flow Control

```
Producer Rate > Consumer Rate?
  → Kafka consumer pause/resume via KafkaListenerEndpointRegistry
  → Circuit breaker OPEN on external service (Resilience4j)
  → Kafka consumer lag alert at 10,000 messages (Prometheus)
  → Auto-scale consumer instances (Kubernetes HPA on lag metric)
```

## 7. New Service: `ihe-gateway-service` (Port 8125)

### Purpose

Dedicated IHE transaction gateway for Oregon HIE interoperability. Implements XDS.b Document Consumer/Source, XCA Initiating/Responding Gateway, and ATNA Secure Node.

### Scope

| Capability | IHE Transaction | Protocol | Priority |
|---|---|---|---|
| Document query | ITI-18 Registry Stored Query | SOAP + REST | Sprint 3 |
| Document retrieve | ITI-43 Retrieve Document Set | SOAP + REST | Sprint 3 |
| Document submit | ITI-41 Provide & Register | SOAP + REST | Sprint 4 |
| Federated query | ITI-38 Cross Gateway Query | SOAP | Sprint 4 |
| Federated retrieve | ITI-39 Cross Gateway Retrieve | SOAP | Sprint 4 |
| Audit emission | ITI-20 Record Audit Event | Syslog/TLS | Sprint 1 |

### Technology

- Java 21 / Spring Boot 3.x (consistent with HDIM stack)
- Apache CXF for SOAP/WS-* support (IHE SOAP transactions)
- HAPI FHIR 7.x for FHIR-based IHE transactions (MHD profile alternative)
- Spring WS for WSDL-first development
- mTLS enforced on all endpoints

### Package Structure

```
com.healthdata.ihegateway/
  config/        # CXF, mTLS, ATNA configuration
  actors/        # IHE actor implementations (DocumentConsumer, DocumentSource, etc.)
  transactions/  # ITI transaction handlers
  audit/         # ATNA audit event generation
  transform/     # XDS metadata ↔ FHIR DocumentReference mapping
  health/        # IHE Connectathon self-test endpoints
```

## 9. Security & Compliance

### 9.1 PHI Classification (Unchanged)

| Integration | PHI Level | Auth | Transport |
|---|---|---|---|
| CoreHive | **NONE** | API key | TLS 1.3 |
| hedis | **LIMITED** | JWT + API key | TLS 1.3 |
| Healthix | **FULL** | mTLS + API key | mTLS (TLS 1.3) |
| IHE Gateway | **FULL** | mTLS (IHE ATNA) | mTLS (TLS 1.3) |

### 9.2 Certificate Management

- CA: HDIM internal CA (or Oregon HIE CA for production)
- Client certs: per-adapter, per-external-system
- Rotation: 90-day automated renewal via cert-manager (K8s)
- Revocation: OCSP stapling enabled on Kong

### 9.3 HIPAA Audit Requirements

Every adapter must log:
- WHO: authenticated principal (API key ID, JWT subject, mTLS CN)
- WHAT: resource type, resource ID, action (CRUD)
- WHEN: ISO-8601 timestamp (NTP-synced, IHE CT compliant)
- WHERE: service name, pod ID, correlation ID
- WHY: business context (care gap closure, measure calculation, document exchange)
- OUTCOME: success/failure, HTTP status, error code
